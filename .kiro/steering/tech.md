## Architecture

### Resource Pattern
- **Orchestrator Pattern**: `conveyor_belt` resource manages sub-resources internally
- **Granular Resources**: `conveyor_belt_lambda` and `conveyor_belt_gateway` for advanced use
- **Manager Pattern**: IAMManager, CloudWatchManager, ParallelManager for AWS operations

### API Gateway Deployment
- **conveyor_belt resource**: Uses **OpenAPI 3.0.1 import** (`PutRestApi` with `mode=overwrite`). One spec per gateway, generated by `OpenAPIGenerator`, deployed by `openapi_deployer.go`. Hash-based change detection skips unchanged gateways.
- **conveyor_belt_gateway resource**: Uses **imperative** API calls (`CreateResource`, `PutMethod`, `PutIntegration`, etc.) via `ParallelRouteProcessor`. Separate code path.

### Parallel Operations
- **ParallelManager**: Semaphore-based concurrency for Lambda operations
- **PackageBuilder**: Concurrent Lambda package building
- **IAMManager**: Parallel IAM reconciliation across all Lambda roles
- **Configurable Concurrency**: `docker_build_concurrency` provider setting

### Change Detection
- **Hash-Based**: Source and config hashes for surgical updates
- **Routes Hash**: Detects changes to routes.tf.rb
- **Lambda Hashes**: Per-Lambda source and config hashes
- **Gateway Hashes**: Per-gateway route configuration hashes
- **OpenAPI Spec Hashes**: Per-gateway spec hash — only redeploys gateways whose spec changed

### Configuration Merge Order
1. Provider defaults (timeout, memory, tags)
2. `lambda_config.shared` settings
3. `lambda_config.{action}` settings
4. Route-derived configuration (tables from routes)
